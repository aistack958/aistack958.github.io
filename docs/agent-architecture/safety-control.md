# 安全与可控性：权限边界、人类监督与幻觉防御

Agent 的自主性是其核心价值，也是其最大风险。一个能够自主调用工具、执行代码、访问数据的智能体，如果缺乏有效的安全约束，可能在一次推理偏差中引发严重后果——删除关键数据、执行危险操作或输出有害内容。安全架构不是 Agent 的附属组件，而是其从实验原型走向生产部署的必要前提。

## 权限边界设计

权限边界定义了 Agent **能够做什么与不能做什么**的明确界限，是安全架构的第一道防线。

### 工具访问控制

Agent 的行动能力源于其可调用的工具集。权限边界设计的第一步是对工具进行分级管理：

- **只读工具（安全级）**：如搜索、查询、读取文件。这些工具不改变外部状态，风险极低，Agent 可自由使用。
- **写入工具（谨慎级）**：如发送消息、修改文件、更新数据库。这些工具改变外部状态，可能产生不可逆影响，应设置操作范围限制与内容审核。
- **危险工具（禁止级/限制级）**：如删除数据、执行系统命令、访问敏感 API。这些工具应被严格限制或默认禁用，仅在明确授权与人类确认后才能使用。

### 操作范围约束

即使允许使用某类工具，也应约束其操作范围：

- **文件系统约束**：限制 Agent 只能在指定目录内读写，禁止访问系统目录与用户隐私目录。
- **网络访问约束**：限制 Agent 只能访问白名单域名，禁止访问内部网络或敏感服务。
- **数据操作约束**：限制数据库操作为 SELECT 为主，INSERT/UPDATE/DELETE 需逐条审核。
- **资源消耗约束**：设置 API 调用频率上限、计算时间上限与费用预算上限，防止 Agent 在错误循环中无限消耗资源。

### 最小权限原则

Agent 的权限配置应遵循最小权限原则——只授予完成当前任务所必需的权限，不多给一项。任务完成后，临时授予的权限应立即收回。这要求权限系统支持动态授权与自动回收机制。

## 人类监督机制

权限边界提供了静态的安全框架，而人类监督机制则处理**动态的安全决策**——那些无法预先设定规则、需要人类判断的关键时刻。

### 关键决策审批（Human-in-the-Loop）

在 Agent 执行流程中设置若干"审批节点"，当 Agent 到达这些节点时暂停执行，将当前决策上下文呈现给人类审核者，等待明确批准后继续。审批节点的设置策略包括：

- **高风险行动前**：任何被标记为"谨慎级"或"限制级"的工具调用前，必须获得人类批准。
- **计划变更时**：当 Agent 因意外情况需要偏离原定计划时，新的执行方案须经人类确认。
- **不确定性阈值触发**：当 Agent 的推理置信度低于设定阈值时，自动转入人类审批模式。

### 观察与干预模式

- **实时观察**：人类可以通过监控面板实时观察 Agent 的推理过程与行动日志，及时发现异常倾向。
- **即时干预**：人类可在任何时刻中断 Agent 执行，修正其当前步骤或注入新的指令。
- **事后审计**：所有 Agent 的推理记录与行动日志被完整保存，支持事后回溯审查与责任追溯。

### 分级监督策略

并非所有任务都需要同等程度的人类监督。根据任务的风险等级设定监督强度：

- **自主执行**：低风险、高确定性任务（如信息查询、文本翻译），Agent 可完全自主执行。
- **关键点审批**：中等风险任务，仅在关键决策节点要求人类确认。
- **全程监督**：高风险任务（如金融交易、医疗诊断辅助），每一步行动都需人类确认后执行。

## 幻觉防御

LLM 的幻觉（Hallucination）——即生成看似合理但实际错误或虚构的内容——是 Agent 安全性的根本威胁。一个自信地输出虚假信息的 Agent，比一个承认不确定性的 Agent 更加危险。

### 事实核查机制

- **内部交叉验证**：要求 Agent 对其关键结论进行自我交叉验证——从不同信息源或不同推理路径独立推导同一结论，只有多条路径一致时才采纳。
- **外部事实核查**：将 Agent 的关键声明提交给独立的核查工具（如搜索引擎、知识库查询），验证其与外部事实的一致性。
- **引用溯源**：要求 Agent 对其事实性声明提供信息来源引用，使得人类可以追溯与验证。

### 置信度标注

要求 Agent 在输出中对每个结论标注置信度等级：

- **高置信（确认）**：有多源验证支撑的事实性结论。
- **中置信（推断）**：基于逻辑推理但缺乏直接验证的结论。
- **低置信（假设）**：基于有限信息或直觉的猜测性结论。

置信度标注使得人类可以优先关注低置信结论，而非被高置信表述掩盖潜在风险。

### 输出过滤与内容安全

即使 Agent 的推理过程合理，其最终输出仍需经过过滤以确保内容安全：

- **敏感信息过滤**：扫描输出中是否包含个人隐私数据、商业机密或受管控信息，如有则自动脱敏或拦截。
- **有害内容检测**：检测输出中是否包含歧视性、煽动性或违反伦理准则的内容。
- **格式与质量校验**：对结构化输出（如代码、SQL、配置文件）进行语法校验与逻辑校验，防止格式错误导致的下游问题。

### 优雅降级策略

当 Agent 无法在安全框架内完成任务时，不应强行给出低质量或高风险的答案，而应**优雅降级**：

- **明确声明局限**：告知用户当前任务超出了 Agent 的能力或安全边界，而非隐瞒困难给出似是而非的答案。
- **提供部分结果**：在能够安全完成的部分给出结果，对不确定部分明确标注为"未验证"。
- **建议人类介入**：当 Agent 判断自身无法安全完成任务时，主动建议将任务转交人类处理。

> 💡 **核心原则**：安全的 Agent 不是"永远不会犯错"的 Agent，而是"犯错时有防护、偏差时有纠正、失控时有兜底"的 Agent。安全架构的目标不是消灭风险，而是将风险控制在可接受范围内，并确保任何失败都有清晰的发现、阻止与恢复机制。

## 本章小结

| 安全层级 | 防护目标 | 核心机制 |
|---------|---------|---------|
| **输入层** | Prompt 注入、恶意输入 | 意图检测、关键词过滤、输入隔离 |
| **权限层** | 工具滥用、越权操作 | 最小权限原则、操作审计、预算上限 |
| **输出层** | 幻觉、有害内容、信息泄露 | 置信度评分、敏感信息过滤、事实校验 |
| **兜底层** | 失控循环、系统性故障 | 优雅降级、人工介入、熔断机制 |

**安全设计原则**：多层叠加 > 单层强控；预防 > 检测 > 恢复；权限最小化 > 权限最大化。
