# 多模态Agent

大多数Agent应用处理的是纯文本——代码、文档、对话记录。但真实世界的信息远不止文字。工厂需要分析设备照片判断故障类型，医生需要结合影像和病历给出诊断，电商需要理解商品图片和用户评论的一致性。多模态Agent就是为这些场景而生——它同时理解视觉和语言信息，在图文之间建立推理桥梁。本文将深入多模态Agent的核心技术和应用构建方法。

## 视觉理解：从像素到语义

视觉理解是多模态Agent的基础能力，但"理解图像"远不止识别图中有什么物体。

### 视觉理解的层次

视觉理解可以分为递进的三个层次：

- **感知层**：识别图像中的基础元素——物体类别、文字内容、颜色分布、空间布局。这是OCR和目标检测的领域，大多数多模态模型已经做得不错。
- **关系层**：理解元素之间的关系——物体A在物体B的左边，图表中的趋势是上升的，界面中的按钮处于禁用状态。关系理解是推理的基础，也是当前最容易出错的层次。
- **推理层**：基于感知和关系进行高级推理——这张电路图说明的是什么工作原理，这个UI截图暗示的设计意图是什么，这张产品照片和描述文字是否存在矛盾。推理层才是多模态Agent真正的价值所在。

### 图像预处理策略

原始图像直接交给模型的效果往往不理想，预处理能显著提升理解质量：

- **关键区域裁剪**：对高分辨率图像，先裁剪关键区域再分别分析。一张1920x1080的网页截图，直接输入会丢失小字细节；裁剪出导航栏、内容区、侧边栏分别分析，每个区域的信息都能被充分捕获。
- **图文分离与组合**：包含大量文字的图像（如幻灯片截图），先做OCR提取文字内容，再结合图像的视觉布局信息。文字信息用文本通道处理精度更高，布局信息用视觉通道处理更准确，两者组合优于单通道。
- **多视角融合**：某些场景需要多个视角的信息融合——同一产品的正面图和侧面图，同一设备的整体图和局部细节图。多模态Agent应该支持多图输入并在推理中关联多图信息。

## 图像与文本协同推理

多模态Agent的核心价值不在单独处理图像或文本，而在两者之间的协同推理。

### 图文对齐与矛盾检测

图文协同最直接的应用是检测图像和文本信息的一致性：

- **电商质检**：商品描述说"红色连衣裙V领"，图片中是蓝色圆领——这种矛盾人工审核容易遗漏，但Agent可以系统化检测。关键是在Prompt中明确要求对照检查："请对照商品描述和商品图片，列出所有不一致之处。"
- **文档完整性验证**：技术文档引用了一个界面截图但截图版本与文字描述不匹配——文档迭代中常见的问题。Agent可以自动对照检查图文版本一致性。
- **合规审查**：广告文案声称的效果和广告图片展示的内容是否一致——这是广告合规审查的核心逻辑，非常适合多模态Agent自动化处理。

### 视觉 grounding

Visual Grounding是让Agent在回答时精确指向图像中的具体区域——"图中的这个接口（指向左侧第三个端口）是USB-C类型"。这种精确指向能力在以下场景中至关重要：

- **故障诊断辅助**：用户上传设备照片描述故障现象，Agent需要能指出"问题出在这个部件（指向图中特定位置），根据外观判断可能是轴承磨损"。
- **教学讲解**：讲解架构图、流程图、电路图时，Agent的回答应该能精确引用图中元素——"这个模块（指向图中标注为Service A的方框）负责处理用户认证请求"。
- **UI审查**：审查界面设计稿时，Agent应该能指出具体位置的问题——"这个按钮（指向右上角区域）的文案和功能描述不一致"。

实现Visual Grounding的技术方案包括：让模型输出坐标标注、使用分割模型预处理关键区域、在图像上叠加可识别的标记索引。当前最实用的方案是在预处理阶段对图像中的关键区域生成编号标记，Agent在回答时用编号引用。

### 跨模态推理链

高级多模态Agent应该能构建跨模态推理链——推理过程中同时引用视觉证据和文本证据：

- **医学影像诊断**：患者影像显示肺部阴影（视觉证据），病历记录患者有长期吸烟史和近期咳嗽症状（文本证据），结合两者推理可能的诊断方向。
- **工程质量检测**：工程照片显示混凝土表面裂缝（视觉证据），施工日志记录该区域浇筑时温度异常（文本证据），推理裂缝成因和修复建议。
- **产品需求分析**：用户反馈截图显示操作困惑（视觉证据），反馈文字描述具体痛点（文本证据），结合两者推导UX改进方案。

跨模态推理链的Prompt设计关键：在系统Prompt中明确要求Agent必须同时引用两个模态的证据支撑结论，不允许只凭单一模态信息做推断。

## 多模态RAG：检索增强的多模态推理

把RAG技术扩展到多模态领域，是构建实用多模态Agent的关键突破。

### 多模态索引构建

传统RAG只索引文本，多模态RAG需要同时索引图像和文本：

- **图像描述生成**：对每张图像生成结构化文本描述——包含物体类别、属性特征、空间关系、图像类型、内容摘要。这些描述用文本检索方法索引，让文本查询能找到相关图像。
- **图文关联索引**：文档中的图像和周围文本有语义关联——图表说明对应的数据趋势，流程图对应文字描述的步骤。索引中要保留这些图文关联关系，检索时可以按关联关系拉取配套的图文组合。
- **视觉特征索引**：对图像提取视觉embedding（用CLIP或SigLIP等模型），建立向量索引。这支持以图搜图——用户上传一张故障设备照片，能检索出知识库中同类设备的历史故障案例。

### 多模态检索策略

查询可能是纯文本、纯图像或图文组合，检索策略需要覆盖所有情况：

- **文本查询检索图像**：通过图像描述索引，用文本查询找到相关图像。用户问"这个接口的接线方式"，检索产品手册中的接线图。
- **图像查询检索文本**：通过视觉特征索引，用图像查询找到描述类似内容的文档。用户上传一张异常截图，检索出描述该异常原因和解决方案的技术文档。
- **图文组合查询检索图文组合**：最复杂的场景——用户提供图文组合查询（如"这张图里的这个部件和手册第三章的描述一致吗"），需要跨模态联合检索找到相关的图文组合内容。

### 检索结果融合

多模态检索返回的可能是文本片段、图像、图文组合的混合结果集。融合策略：

- **相关性排序**：用交叉编码器对混合结果集统一重排，确保最相关的图文组合排在前面。
- **模态互补**：如果检索到的文本片段缺少关键视觉信息，补充检索相关图像；如果检索到的图像缺少文字解释，补充检索相关文本描述。
- **上下文窗口管理**：图像token消耗远大于文本。多模态RAG必须精细管理上下文窗口——限制图像数量但保证每张图像有足够的文字上下文支撑，避免上下文窗口被少数几张高清图像占满。

## 应用场景与架构选择

不同多模态应用场景对Agent架构有不同要求：

- **文档理解型**（如PDF报告分析、技术手册问答）：以文本为主、图像为辅。架构上用文本RAG为主通道，视觉理解在需要时补充调用。
- **视觉分析型**（如工厂质检、医学影像辅助）：以图像为主、文本为辅。架构上先做视觉理解提取关键信息，再用文本RAG检索相关知识和历史案例辅助推理。
- **图文协同型**（如电商内容审核、广告合规检查）：图像和文本同等重要。架构上必须双通道并行处理，在推理阶段做跨模态对齐和矛盾检测。

选择架构时不要试图做通用万能方案——针对场景的专用架构更高效、更可靠、更容易评估和优化。

多模态Agent代表了AI应用从文本世界走向真实世界的方向。构建多模态Agent的挑战不在模型能力本身，而在如何让视觉和语言两种信息在推理中有效协同，以及如何在检索增强的框架中整合多模态数据。掌握这些关键技术，才能真正构建出理解真实世界的智能Agent。

## 本章小结

多模态 Agent 三层能力：
1. **感知层**：图像识别、OCR、物体检测
2. **关系层**：视觉-文本关联、空间关系理解
3. **推理层**：跨模态推理、多模态 RAG、视觉问答

**应用场景**：文档理解、电商商品分析、医学影像、自动驾驶感知。
