# 数据分析Agent

数据分析Agent的目标是把数据分析从专业技能变为通用能力。让产品经理用自然语言就能查询用户行为数据，让运营人员一句话就能生成转化漏斗报告，让管理者直接问"上个月华东区域哪个产品线增长最快"就得到完整分析结果。本文将从SQL生成到洞察提取，覆盖数据分析Agent的完整技术栈。

## 自然语言查询数据：从意图到SQL

自然语言到SQL的转换（Text-to-SQL）是数据分析Agent的基础能力，但也是最容易出问题的环节。

### 查询意图理解

用户说"看看最近的销售情况"，这句话包含大量隐含信息：哪个时间范围算"最近"、销售情况看哪些指标（金额还是订单量还是增长率）、数据粒度是按天还是按月。Agent必须通过以下策略解析这些隐含信息：

- **数据库元信息注入**：Agent需要了解数据库的完整元信息——表结构、字段含义、表间关系、常见查询模式。这些元信息应该在Agent的系统上下文中持续可用，而不是每次查询时重新收集。
- **意图澄清对话**：对于模糊查询，Agent不应该猜测用户意图然后生成可能错误的SQL，而是通过简短对话澄清关键参数。"最近"是近7天还是近30天？"销售情况"看总金额还是分产品线拆解？一两轮澄清就能大幅提升SQL准确率。
- **查询模式匹配**：大多数业务查询符合可识别的模式——趋势分析、对比分析、排名分析、分布分析、漏斗分析。识别查询模式后，可以套用对应的SQL模板骨架，只需要填充具体的字段和条件。

### SQL生成的质量控制

生成的SQL必须正确，这不是可选要求而是底线。实战中的质量控制策略：

- **Schema约束严格化**：给模型的Prompt中不仅要列出表结构，还要标注字段类型、枚举值、外键关系、常见取值范围。这些约束信息让模型减少类型错误、字段名误用、条件值偏差。
- **SQL验证层**：生成SQL后不直接执行，而是先通过验证层——语法检查、字段存在性校验、类型兼容性检查。验证失败则反馈给模型重新生成，而不是带着错误执行。
- **复杂查询拆解**：复杂分析需求不要试图用一条巨型SQL解决。把复杂查询拆解为多个步骤——先查基础数据，再做聚合计算，最后组合结果。每步独立验证，整体逻辑更可控。

## 图表绘制：让数据可视化

数据分析的价值不只在于查出数字，更在于让数字变成可理解的故事。图表绘制是数据到故事的桥梁。

### 智能图表选择

不同数据适合不同的可视化形式，Agent应该根据数据特征自动选择图表类型：

- **趋势数据**：时间序列指标用折线图，多条趋势对比用多线折线图或面积图。
- **对比数据**：分组指标对比用柱状图，比例对比用饼图或环形图，多维对比用雷达图。
- **分布数据**：连续值分布用直方图，离散值分布用条形图，地理分布用地图热力图。
- **关联数据**：两个指标的关联性用散点图，多维关联用气泡图或热力矩阵。

图表选择的逻辑不应该硬编码规则，而是让Agent根据查询意图和数据结构推理选择。系统Prompt中提供图表选择指南，让模型在生成分析结果时同步决定可视化方案。

### 图表生成技术方案

主流技术方案有两种：

- **代码生成方案**：Agent生成Python代码（matplotlib/plotly/seaborn），在沙箱中执行生成图表图片。优势是灵活度高，可以定制任何可视化细节；劣势是需要沙箱执行环境，图表样式调优需要迭代。
- **声明式配置方案**：Agent生成图表配置JSON（ECharts/Vega-Lite规范），前端直接渲染。优势是无需后端执行环境，渲染速度快，交互性好；劣势是灵活度受限于配置规范支持的范围。

实战推荐混合方案：常规分析场景用声明式配置快速生成标准图表，定制化需求用代码生成方案实现特殊可视化。

## 洞察提取：超越数字的解读

查询数据和绘制图表只是展示事实，洞察提取才是分析的灵魂。Agent应该从数据中主动发现值得关注的模式、异常和机会。

### 自动异常检测

Agent在返回数据结果时，应该同步执行异常检测：

- **阈值异常**：指标超出历史正常范围（如日活突然下降30%），主动标注并追问原因。
- **趋势异常**：指标趋势发生拐点（如增长率从正转负），标注拐点位置和可能触发因素。
- **分布异常**：数据分布出现不应存在的模式（如某个地区订单量异常集中），提示可能的业务问题或数据质量问题。

异常检测不需要复杂模型——基于历史数据的统计区间、环比同比的简单对比、分布形状的基础判断就够了。关键是Agent要主动做这些检测，而不是只返回原始数据等用户自己发现异常。

### 对比与归因分析

洞察不只是发现异常，还要解释异常：

- **维度拆解**：总量下降时，自动按维度拆解找出哪个维度贡献了主要下降——是哪个产品线、哪个地区、哪个渠道的问题。
- **归因推理**：基于时间序列相关性推理可能的影响因素。日活下降和某个功能上线时间吻合，提示可能是该功能导致的问题。这不是确定性因果推理，而是辅助假设生成。
- **对比基准选择**：洞察需要有基准才能判断好坏。Agent应该自动选择合适的对比基准——同比（去年同期）、环比（上月）、目标值（预算/计划）、同类参照（行业均值）。

### 洞察呈现格式

洞察应该以结构化格式呈现，而不是混在冗长的自然语言段落中：

- **摘要优先**：先给一两句话的核心结论，再展开详细分析。忙碌的管理者需要30秒内抓住要点。
- **证据链接**：每个洞察结论都应该附链接到支撑数据——点击结论就能看到对应的图表和原始查询结果。
- **建议行动**：洞察最终要导向行动。Agent应该在分析结论后给出建议——"华东区域增长放缓可能与竞品X的促销活动有关，建议监测竞品动态并评估是否需要针对性响应。"

## 自然语言交互的体验设计

数据分析Agent的交互体验需要特别关注：

- **渐进式揭示**：先给核心结论，用户追问时再展示细节。不要在第一次回答时就输出一个包含五张图表和三段分析的完整报告——信息过载反而降低理解效率。
- **交互式修正**：当Agent对查询意图的理解可能有偏差时，先展示理解后的查询逻辑，让用户确认或修正，再执行查询。"你问的是近30天各产品线的销售金额对比，我准备查询orders表按product_category分组聚合近30天的total_amount，这样对吗？"
- **分析延续**：每次分析结果后，Agent应该主动建议后续分析方向——"数据显示华东区域增长放缓，你想进一步看华东各城市的细分数据，还是看同期其他区域的表现作为对比？"

数据分析Agent让数据驱动决策从"有能力的人才能做的事"变成"所有人都能做的事"。但这个转变的关键不只是技术实现，更是交互设计——让用户在自然对话中逐步深入数据，而不是被一次性淹没在分析结果中。
