# Agent 相关面试题解析

本章精选 AI Agent 岗位的高频面试题，每道题给出考点分析、参考回答框架和加分要点。这些题目覆盖了 Agent 架构设计、工具调用、检索增强、效果评估、协议标准等核心领域。

## 1. 如何设计一个多Agent系统？

**考点**：多Agent协作模式理解、架构拆分能力、通信机制设计

**参考回答框架**：

多Agent系统的设计首先要明确协作模式。常见模式有三种：**层级式**（一个中央调度Agent分配任务给子Agent）、**协作式**（多个Agent平等协作，通过共享记忆或消息传递协调）、**竞争式**（多个Agent各自独立尝试解决问题，择优选用）。

以代码生成场景为例，层级式架构中调度Agent接收用户需求，拆解为"需求分析→代码编写→代码审查"三个子任务，分别交给分析Agent、编码Agent、审查Agent执行。协作式架构中各Agent共享一个工作记忆区，编码Agent写完代码后自动通知审查Agent，审查Agent的反馈写入共享记忆供编码Agent参考。

关键设计决策包括：Agent间通信方式（共享记忆 vs 消息队列）、任务分配策略（静态分工 vs 动态路由）、冲突解决机制（当两个Agent给出矛盾建议时如何裁决）。还要考虑系统复杂度与效果的平衡——2-3个Agent的简单协作往往比7-8个Agent的复杂编排更实用。

**加分要点**：提到实际工程中的踩坑经验，如"多Agent系统的调试成本远高于单Agent，建议先用单Agent验证可行性再考虑多Agent拆分"。

## 2. RAG 和微调，什么时候选哪个？

**考点**：对两种LLM增强方式的本质理解、工程决策判断力

**参考回答框架**：

RAG和微调解决的是不同问题。**RAG**解决的是"知识注入"问题——让模型访问它训练数据中没有的外部信息。**微调**解决的是"行为塑造"问题——让模型在特定任务上表现得更专业、更稳定。

选择决策的核心依据是：如果问题主要是"模型不知道某些信息"（如公司内部文档、最新新闻），选RAG；如果问题主要是"模型在该任务上表现不够好"（如遵循特定输出格式、采用特定推理风格），选微调。

实际项目中两者常常组合使用。例如客服Agent场景：先用微调让模型学会客服对话风格和标准回复格式，再用RAG让模型能查询产品手册和用户订单信息。单独只用一种往往不够——纯RAG的模型可能检索到信息但无法以正确风格呈现，纯微调的模型可能风格对了但缺少最新信息。

成本维度也要考虑：RAG的运维成本主要是向量数据库和检索管道，微调的成本主要是数据标注和GPU训练资源。对于快速迭代的产品，RAG的灵活性更高（更新知识只需更新数据库），微调的迭代周期更长。

**加分要点**：提到"渐进策略"——先用RAG快速验证效果，确认方向正确后再考虑微调做行为优化，避免过早投入微调成本。

## 3. 如何评估 Agent 的效果？

**考点**：评估方法论理解、指标设计能力、对Agent评估特殊性的认知

**参考回答框架**：

Agent评估比传统LLM评估复杂得多，因为Agent的输出不只是一段文本，而是包含推理过程、工具调用、多步决策的完整行为轨迹。

核心评估维度有四个：**任务完成率**（Agent是否最终解决了用户的问题）、**步骤效率**（Agent用了多少步才完成任务，过多的步骤说明规划能力不足）、**工具使用准确率**（Agent选择和调用工具是否正确）、**成本效率**（完成任务消耗的token量和API调用次数）。

具体评估方法：对于确定性任务（如信息查询），可以用自动化测试集对比Agent输出与标准答案。对于开放性任务（如创意写作辅助），需要人工评估或用LLM-as-judge方法。重要的是评估Agent的**鲁棒性**——在模糊输入、工具失败、信息不足等边界情况下Agent是否还能合理应对，而非只在理想场景下表现好。

一个实用建议是建立"阶梯测试集"：从简单单工具调用→中等多工具组合→复杂多步推理，逐级测试Agent能力边界，找到它的能力上限和退化点。

**加分要点**：提到评估不应只看单次结果，还要看Agent在多次运行中的稳定性（同一个问题跑5次，几次能得到正确答案）。

## 4. MCP 协议的价值是什么？

**考点**：对Agent工具调用标准化的理解、协议设计的工程思维

**参考回答框架**：

MCP（Model Context Protocol）的核心价值是**标准化Agent与外部工具的交互接口**。类比HTTP协议标准化了Web应用间的通信，MCP标准化了LLM Agent与工具/数据源的通信。

没有MCP时，每个工具都需要针对每个LLM写专门的适配层——LangChain的工具定义格式、OpenAI的Function Calling格式、各Agent框架的自有格式各不相同，工具开发者需要适配多个平台。有了MCP，工具端只需实现一个标准协议的MCP Server，任何支持MCP的Agent框架都能直接调用，大幅降低了工具集成的开发和维护成本。

MCP协议的设计包含几个关键组件：**Tools**（可调用的函数接口）、**Resources**（可读取的数据源）、**Prompts**（预定义的提示模板）。这种分类让Agent能清晰地理解每个MCP Server提供什么能力，而不是面对一堆未分类的API端点。

从Agent开发者角度看，MCP让工具生态从"每个Agent框架各自建工具库"变为"共享工具生态"，类似于npm之于Node.js——你不再需要自己写每个工具的适配器，而是直接接入已有的MCP Server。

**加分要点**：提到MCP当前仍处于早期阶段，实际采用中的挑战包括协议稳定性、工具质量参差、安全沙箱机制不成熟等，展现你对新技术的理性判断。

## 5. Agent 的记忆机制如何设计？

**考点**：对Agent长期/短期记忆的理解、状态管理能力

**参考回答框架**：

Agent记忆分为**短期记忆**和**长期记忆**。短期记忆是当前对话的上下文窗口，受LLM的token限制约束，通常通过滑动窗口或摘要压缩来管理。长期记忆是跨对话的知识积累，常见实现方式包括向量数据库存储历史交互摘要、结构化数据库存储用户偏好和知识图谱。

设计关键是记忆的**检索和更新策略**。检索时要平衡召回率和相关性——过多无关记忆会干扰Agent的当前决策，太少记忆又让Agent缺乏上下文。更新时要考虑遗忘机制——不是所有历史信息都值得长期保留，要有选择性地提炼和压缩。

一个实用模式是"三层记忆架构"：工作记忆（当前对话上下文）→会话记忆（近期多次对话的摘要）→长期记忆（提炼的结构化知识）。Agent在不同场景下访问不同层级的记忆，既保证当前任务的专注性，又不丢失长期积累。

## 6. 如何处理Agent的工具调用失败？

**考点**：鲁棒性设计思维、异常处理能力

**参考回答框架**：

工具调用失败是Agent运行中的常态而非例外。处理策略包括：**重试机制**（同一工具短暂失败后重试，设置最大重试次数）、**降级策略**（主工具失败后切换到备用工具或简化方案）、**反馈学习**（将失败信息反馈给LLM，让它重新规划和选择替代行动）、**优雅降级**（所有工具都失败时，Agent应坦诚告知用户当前能力受限而非胡编答案）。

关键设计原则是**不要把工具失败当作异常情况**，而是当作Agent正常运行中需要处理的常规事件。在Agent的提示词中明确告知"工具可能返回错误结果或失败，你应该根据错误信息调整下一步行动"。

## 7. Agent 的安全边界如何设计？

**考点**：对Agent安全风险的理解、防护机制设计

**参考回答框架**：

Agent安全的核心风险包括：**Prompt注入**（恶意输入操纵Agent行为）、**工具滥用**（Agent被诱导调用危险工具）、**信息泄露**（Agent输出敏感数据）、**权限越界**（Agent执行超出预期范围的操作）。

防护机制应多层叠加：输入层做意图检测和恶意prompt识别，工具层做权限分级和操作审计，输出层做敏感信息过滤，整体层面设置行动预算（最大步数、最大token消耗）防止失控循环。

## 8. 为什么选择 ReAct 而非纯规划式 Agent？

**考点**：对不同Agent范式的理解、架构选择判断力

**参考回答框架**：

ReAct（思考-行动-观察循环）的核心优势是**动态适应**——Agent可以根据工具返回的实际结果调整下一步行动，而非执行一个可能基于错误假设的固定计划。纯规划式Agent先生成完整计划再逐步执行，在信息不完整或环境变化的场景中容易偏离实际。

选择依据：当任务需要与外部环境频繁交互、信息逐步获取时，选ReAct；当任务步骤确定性强、执行过程不可中断时，纯规划式效率更高。实际中两者常混合使用——先做粗粒度规划，再在每个步骤内用ReAct细化执行。

## 9. 如何降低 Agent 的推理成本？

**考点**：工程优化思维、成本意识

**参考回答框架**：

主要优化方向：**模型选择分层**（简单判断用小模型，复杂推理用大模型）、**缓存策略**（相似查询复用历史结果、工具返回结果缓存）、**提示压缩**（精简系统提示、用摘要替代完整历史）、**工具结果精炼**（工具返回的大量数据先做提取和压缩再送入LLM）、**提前终止**（设置置信度阈值，Agent确信已获得答案时提前结束循环）。

## 10. Agent 在生产环境中的主要挑战是什么？

**考点**：工程落地经验、对生产级要求的认知

**参考回答框架**：

核心挑战包括：**延迟管理**（多步Agent推理的响应时间远超单次LLM调用）、**稳定性保障**（LLM输出的不确定性导致Agent行为不可预测）、**可观测性**（Agent的决策链路复杂，排查问题困难）、**成本控制**（多步推理的token消耗是乘法级增长）、**安全合规**（Agent自主调用外部工具带来权限和数据安全风险）。每个挑战都需要专门的工程方案，而非单纯靠更好的LLM来解决。
