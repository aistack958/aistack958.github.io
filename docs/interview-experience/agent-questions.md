# Agent 面试题解析

本章按模块整理 AI Agent 岗位的高频面试题，每道题给出考点分析、参考回答框架和加分要点。题目覆盖 Agent 架构设计、开发框架、RAG 检索、向量数据库、工程实践与应用开发等核心领域，建议结合自身项目经验灵活组织回答。

## 一、Agent 基础与架构设计

### 面试题 1：如何设计一个多 Agent 系统？

**考点**：多 Agent 协作模式理解、架构拆分能力、通信机制设计

**参考回答框架**：

多 Agent 系统的设计首先要明确协作模式。常见模式有三种：**层级式**（一个中央调度 Agent 分配任务给子 Agent）、**协作式**（多个 Agent 平等协作，通过共享记忆或消息传递协调）、**竞争式**（多个 Agent 各自独立尝试解决问题，择优选用）。

以代码生成场景为例，层级式架构中调度 Agent 接收用户需求，拆解为"需求分析 → 代码编写 → 代码审查"三个子任务，分别交给分析 Agent、编码 Agent、审查 Agent 执行。协作式架构中各 Agent 共享一个工作记忆区，编码 Agent 写完代码后自动通知审查 Agent，审查 Agent 的反馈写入共享记忆供编码 Agent 参考。

关键设计决策包括：Agent 间通信方式（共享记忆 vs 消息队列）、任务分配策略（静态分工 vs 动态路由）、冲突解决机制（当两个 Agent 给出矛盾建议时如何裁决）。还要考虑系统复杂度与效果的平衡——2-3 个 Agent 的简单协作往往比 7-8 个 Agent 的复杂编排更实用。

**加分要点**：提到实际工程中的踩坑经验，如"多 Agent 系统的调试成本远高于单 Agent，建议先用单 Agent 验证可行性再考虑多 Agent 拆分"。

### 面试题 2：Agent 的记忆机制如何设计？

**考点**：对 Agent 长期/短期记忆的理解、状态管理能力

**参考回答框架**：

Agent 记忆分为**短期记忆**和**长期记忆**。短期记忆是当前对话的上下文窗口，受 LLM 的 token 限制约束，通常通过滑动窗口或摘要压缩来管理。长期记忆是跨对话的知识积累，常见实现方式包括向量数据库存储历史交互摘要、结构化数据库存储用户偏好和知识图谱。

设计关键是记忆的**检索和更新策略**。检索时要平衡召回率和相关性——过多无关记忆会干扰 Agent 的当前决策，太少记忆又让 Agent 缺乏上下文。更新时要考虑遗忘机制——不是所有历史信息都值得长期保留，要有选择性地提炼和压缩。

一个实用模式是"三层记忆架构"：工作记忆（当前对话上下文）→ 会话记忆（近期多次对话的摘要）→ 长期记忆（提炼的结构化知识）。Agent 在不同场景下访问不同层级的记忆，既保证当前任务的专注性，又不丢失长期积累。

**加分要点**：结合具体场景说明记忆检索的实现细节，如"用 Embedding 相似度检索长期记忆时，要对用户当前 query 做意图扩展，避免关键词不完全匹配导致的漏召回"。

### 面试题 3：为什么选择 ReAct 而非纯规划式 Agent？

**考点**：对不同 Agent 范式的理解、架构选择判断力

**参考回答框架**：

ReAct（思考-行动-观察循环）的核心优势是**动态适应**——Agent 可以根据工具返回的实际结果调整下一步行动，而非执行一个可能基于错误假设的固定计划。纯规划式 Agent 先生成完整计划再逐步执行，在信息不完整或环境变化的场景中容易偏离实际。

选择依据：当任务需要与外部环境频繁交互、信息逐步获取时，选 ReAct；当任务步骤确定性强、执行过程不可中断时，纯规划式效率更高。实际中两者常混合使用——先做粗粒度规划，再在每个步骤内用 ReAct 细化执行。

**加分要点**：提到 ReAct 的推理过程本身就是可观测的调试线索，而纯规划式 Agent 一旦偏离计划，排查根因更困难。

### 面试题 4：Agent 的安全边界如何设计？

**考点**：对 Agent 安全风险的理解、防护机制设计

**参考回答框架**：

Agent 安全的核心风险包括：**Prompt 注入**（恶意输入操纵 Agent 行为）、**工具滥用**（Agent 被诱导调用危险工具）、**信息泄露**（Agent 输出敏感数据）、**权限越界**（Agent 执行超出预期范围的操作）。

防护机制应多层叠加：输入层做意图检测和恶意 prompt 识别，工具层做权限分级和操作审计，输出层做敏感信息过滤，整体层面设置行动预算（最大步数、最大 token 消耗）防止失控循环。

**加分要点**：提到"最小权限原则"——每个工具只暴露必要的操作，Agent 的默认状态应该是"能做的不多"而非"限制的不多"。

## 二、Agent 开发框架与协议

### 面试题 5：MCP 协议的价值是什么？

**考点**：对 Agent 工具调用标准化的理解、协议设计的工程思维

**参考回答框架**：

MCP（Model Context Protocol）的核心价值是**标准化 Agent 与外部工具的交互接口**。类比 HTTP 协议标准化了 Web 应用间的通信，MCP 标准化了 LLM Agent 与工具/数据源的通信。

没有 MCP 时，每个工具都需要针对每个 LLM 写专门的适配层——LangChain 的工具定义格式、OpenAI 的 Function Calling 格式、各 Agent 框架的自有格式各不相同，工具开发者需要适配多个平台。有了 MCP，工具端只需实现一个标准协议的 MCP Server，任何支持 MCP 的 Agent 框架都能直接调用，大幅降低了工具集成的开发和维护成本。

MCP 协议的设计包含几个关键组件：**Tools**（可调用的函数接口）、**Resources**（可读取的数据源）、**Prompts**（预定义的提示模板）。这种分类让 Agent 能清晰地理解每个 MCP Server 提供什么能力。

**加分要点**：提到 MCP 当前仍处于早期阶段，实际采用中的挑战包括协议稳定性、工具质量参差、安全沙箱机制不成熟等，展现你对新技术的理性判断。

### 面试题 6：OpenAI Function Calling 和 Claude Tool Use 有什么区别？

**考点**：对主流工具调用协议的深入理解、工程适配能力

**参考回答框架**：

两者核心差异有三点：

1. **工具结果的角色不同**：OpenAI 用 `role: "tool"`，Claude 用 `role: "user"`。这意味着在 Claude 中，工具结果是"用户消息"的一部分，对话历史管理逻辑需要适配
2. **参数是对象还是字符串**：OpenAI 的 `arguments` 是字符串需要 `JSON.parse()`，Claude 的 `input` 是对象可以直接使用。Claude 的设计更安全——避免了模型生成非法 JSON 字符串的问题
3. **混合输出能力**：Claude 可以在同一条消息中既输出文本解释又发起工具调用，这让 Agent 的可观测性更好。OpenAI 在调用工具时 `content` 为 null，丢失了推理解释

从工程角度看，如果需要同时兼容两者，建议在工具执行层之上封装一个统一的抽象层，将不同协议的差异隔离在适配器内部，上层业务代码不感知底层是 OpenAI 还是 Claude。

**加分要点**：提到多工具场景的分层路由策略——当工具超过 20 个时，先用一个"路由工具"选择类别，再暴露具体工具，提升模型选择准确率。

### 面试题 7：Skill 系统与 MCP 是什么关系？

**考点**：对新兴 Agent 能力抽象层的理解、架构分层思维

**参考回答框架**：

Skill 和 MCP 是**互补而非竞争**的关系：

- **MCP 解决"连接"问题**：标准化 Agent 与外部工具/数据源的通信协议
- **Skill 解决"能力"问题**：封装完整的业务流程和决策逻辑

**类比**：MCP 是 USB 接口标准，Skill 是插在上面的外设驱动程序——接口让设备能连上电脑，驱动程序让设备能真正工作。

在工程实践中，一个 Skill 通常**内部使用 MCP** 来连接底层工具。例如"旅行规划 Skill"内部通过 MCP 连接航班查询、酒店预订、天气查询等 Server，但对外暴露的是一个完整的"规划旅行"能力——它知道何时调用哪些工具、如何处理中间结果、如何在失败时降级。

**加分要点**：提到 Skill 生态的治理挑战——社区贡献的 Skill 中有相当比例存在安全漏洞（Prompt 注入、权限越界等），生产环境使用需经过沙箱测试和代码审计。

## 三、RAG 与向量检索

### 面试题 8：RAG 和微调，什么时候选哪个？

**考点**：对两种 LLM 增强方式的本质理解、工程决策判断力

**参考回答框架**：

RAG 和微调解决的是不同问题。**RAG**解决的是"知识注入"问题——让模型访问它训练数据中没有的外部信息。**微调**解决的是"行为塑造"问题——让模型在特定任务上表现得更专业、更稳定。

选择决策的核心依据是：如果问题主要是"模型不知道某些信息"（如公司内部文档、最新新闻），选 RAG；如果问题主要是"模型在该任务上表现不够好"（如遵循特定输出格式、采用特定推理风格），选微调。

实际项目中两者常常组合使用。例如客服 Agent 场景：先用微调让模型学会客服对话风格和标准回复格式，再用 RAG 让模型能查询产品手册和用户订单信息。单独只用一种往往不够——纯 RAG 的模型可能检索到信息但无法以正确风格呈现，纯微调的模型可能风格对了但缺少最新信息。

成本维度也要考虑：RAG 的运维成本主要是向量数据库和检索管道，微调的成本主要是数据标注和 GPU 训练资源。对于快速迭代的产品，RAG 的灵活性更高。

**加分要点**：提到"渐进策略"——先用 RAG 快速验证效果，确认方向正确后再考虑微调做行为优化，避免过早投入微调成本。

### 面试题 9：RAG 系统中如何提升检索质量？

**考点**：RAG 工程化能力、检索优化方法论

**参考回答框架**：

提升 RAG 检索质量是一个系统工程，核心环节包括：

**分块策略**：块大小直接影响检索质量。经验值 512-1024 tokens 为通用起点，法律/合同等长文本场景可用 1500-2000，问答对等短文本场景用 200-400。更关键的是分块边界——不要在段落中间硬切，应按语义边界分割，可用 `semantic-chunking` 策略计算相邻句子的 Embedding 相似度，低于阈值处切分。

**混合检索**：结合向量检索与关键词检索（BM25），取各自优势。向量检索擅长语义匹配，BM25 擅长精确匹配。实际做法是分别检索后做 RRF（Reciprocal Rank Fusion）融合排序，公式为 `score = sum(1 / (k + rank_i))`，k 通常取 60。

**重排序（Reranking）**：向量检索的 Top-K 结果往往包含"看起来相关但实际无关"的段落。用交叉编码器（Cross-Encoder）对 query 与每个候选段落做精细相关性打分。推荐先用向量检索召回 20-50 条候选，再用 Reranker 精选 5-10 条送入 LLM。

**元数据过滤**：纯向量检索不考虑文档的时效性、权限、类别等元数据。在向量数据库中存储元数据字段，检索时附加过滤条件（如 `date > 2024-01-01 AND permission = "public"`），避免召回过时或无权访问的文档。

**加分要点**：提到评估指标——不要凭感觉优化，要建立 Recall@K、Precision@K、NDCG@K 等量化指标，用 100-500 条真实查询构成评估基准集，每次优化后跑自动化评测对比。

### 面试题 10：Embedding 模型如何选型？

**考点**：对 Embedding 技术的理解、模型选型能力

**参考回答框架**：

Embedding 模型的选择直接影响检索质量，需从三个维度评估：

| 维度 | 关注点 |
|------|--------|
| **语义理解能力** | 同义词、否定、多义词的处理质量 |
| **维度与性能** | 维度越高表达力越强，但索引和检索成本也越高 |
| **多语言支持** | 中英混合文档场景下是否兼顾两种语言 |

**常见选项**：

- **OpenAI text-embedding-3-large**：3072 维，语义理解强，适合中英文混合场景，但成本较高
- **text-embedding-3-small**：1536 维，性价比好，大多数场景够用
- **BGE-M3（BAAI）**：开源模型，支持多语言和多粒度检索，适合私有化部署
- **GTE-Qwen2**：阿里开源，中文表现优异，适合中文为主的知识库

**选型建议**：先在业务数据集上做小规模召回率测试（50-200 条查询），不要仅看模型排行榜——排行榜数据集与你的业务数据分布很可能不同。

**加分要点**：提到 Embedding 微调——如果业务领域非常垂直（如医疗、法律），通用 Embedding 可能无法捕捉领域术语的细微差别，此时在领域语料上微调 Embedding 模型往往比换更大的通用模型更有效。

## 四、向量数据库与工程基础

### 面试题 11：向量数据库选型时需要考虑哪些因素？

**考点**：对向量数据库生态的理解、基础设施选型能力

**参考回答框架**：

向量数据库选型需考虑以下因素：

**数据规模与性能**：百万级文档可用开源方案（Milvus、Weaviate、Qdrant），亿级以上或高并发场景考虑托管服务（Pinecone、Zilliz Cloud）。要实测写入吞吐和查询延迟，不要只看官方 benchmark。

**混合检索支持**：生产环境几乎都需要"向量相似度 + 关键词过滤"的组合查询，确保选型支持 metadata filtering 和 hybrid search。Milvus、Pinecone、Weaviate 都支持，但语法和性能差异较大。

**部署与运维**：私有化部署选 Milvus（K8s 生态成熟）或 Qdrant（轻量易部署）；无运维团队选托管服务。要考虑数据备份、索引重建、版本升级等运维成本。

**成本模型**：开源方案主要成本是服务器资源，托管服务按数据量+查询量计费。小规模场景托管服务更省心，大规模场景自托管成本更低。

**加分要点**：提到"不要过度设计"——很多项目早期用 pgvector（PostgreSQL 插件）就能满足需求，等数据量到百万级再迁移到专用向量数据库，避免过早引入运维复杂度。

### 面试题 12：如何处理 Agent 的工具调用失败？

**考点**：鲁棒性设计思维、异常处理能力

**参考回答框架**：

工具调用失败是 Agent 运行中的常态而非例外。处理策略包括：

- **重试机制**：同一工具短暂失败后重试，设置最大重试次数（通常 3 次），对可恢复错误（网络超时、429 限流）做指数退避重试
- **降级策略**：主工具失败后切换到备用工具或简化方案（如实时查询失败时返回缓存数据并标注"数据可能非最新"）
- **反馈学习**：将失败信息反馈给 LLM，让它重新规划和选择替代行动
- **优雅降级**：所有工具都失败时，Agent 应坦诚告知用户当前能力受限而非胡编答案

关键设计原则是**不要把工具失败当作异常情况**，而是当作 Agent 正常运行中需要处理的常规事件。在 Agent 的提示词中明确告知"工具可能返回错误结果或失败，你应该根据错误信息调整下一步行动"。

**加分要点**：提到错误分级体系——Level 0（可忽略，部分数据缺失）→ Level 1（可重试，暂时性错误）→ Level 2（需要替代，工具不可用）→ Level 3（需要人工介入，权限/冲突问题），不同级别对应不同的 Agent 处理策略。

## 五、Agent 工程实践

### 面试题 13：如何评估 Agent 的效果？

**考点**：评估方法论理解、指标设计能力、对 Agent 评估特殊性的认知

**参考回答框架**：

Agent 评估比传统 LLM 评估复杂得多，因为 Agent 的输出不只是一段文本，而是包含推理过程、工具调用、多步决策的完整行为轨迹。

核心评估维度有四个：**任务完成率**（Agent 是否最终解决了用户的问题）、**步骤效率**（Agent 用了多少步才完成任务，过多的步骤说明规划能力不足）、**工具使用准确率**（Agent 选择和调用工具是否正确）、**成本效率**（完成任务消耗的 token 量和 API 调用次数）。

具体评估方法：对于确定性任务（如信息查询），可以用自动化测试集对比 Agent 输出与标准答案。对于开放性任务（如创意写作辅助），需要人工评估或用 LLM-as-judge 方法。重要的是评估 Agent 的**鲁棒性**——在模糊输入、工具失败、信息不足等边界情况下 Agent 是否还能合理应对。

一个实用建议是建立"阶梯测试集"：从简单单工具调用 → 中等多工具组合 → 复杂多步推理，逐级测试 Agent 能力边界，找到它的能力上限和退化点。

**加分要点**：提到评估不应只看单次结果，还要看 Agent 在多次运行中的稳定性（同一个问题跑 5 次，几次能得到正确答案），以及多轮对话中的上下文一致性。

### 面试题 14：Agent 在生产环境中的主要挑战是什么？

**考点**：工程落地经验、对生产级要求的认知

**参考回答框架**：

核心挑战包括：

- **延迟管理**：多步 Agent 推理的响应时间远超单次 LLM 调用，用户可接受的响应时间通常在 1-3 秒，而复杂 Agent 任务可能需要 10-30 秒。解决方案包括流式输出（让用户看到中间进展）、异步执行（对复杂任务走后台处理+通知机制）、步骤并行化（能并行的工具调用同时发起）
- **稳定性保障**：LLM 输出的不确定性导致 Agent 行为不可预测，同一问题多次运行可能得到不同结果。解决方案包括温度参数调低、输出格式强制约束（JSON Schema）、关键节点增加人工确认
- **可观测性**：Agent 的决策链路复杂，排查问题困难。需要记录完整的 Thought-Action-Observation 轨迹，每个工具调用的输入输出，以及每次 LLM 调用的上下文
- **成本控制**：多步推理的 token 消耗是乘法级增长。需要模型分层（简单判断用小模型）、缓存策略、提前终止机制
- **安全合规**：Agent 自主调用外部工具带来权限和数据安全风险，需要审计日志、权限分级、敏感数据脱敏

**加分要点**：提到这些挑战"不是单纯靠更好的 LLM 来解决"——再强的模型也需要工程层面的延迟优化、稳定性保障和可观测性建设。

### 面试题 15：如何降低 Agent 的推理成本？

**考点**：工程优化思维、成本意识

**参考回答框架**：

主要优化方向：

- **模型选择分层**：简单判断（如意图分类、工具选择）用小模型（GPT-3.5 级别），复杂推理（如代码生成、多步规划）用大模型。通过路由层根据任务复杂度动态选择模型
- **缓存策略**：相似查询复用历史结果（用向量相似度判断"这个问题是否问过"）、工具返回结果缓存（如天气查询结果 10 分钟内有效）
- **提示压缩**：精简系统提示（去掉冗余描述）、用摘要替代完整历史（只保留关键决策点和当前上下文）
- **工具结果精炼**：工具返回的大量数据先做提取和压缩再送入 LLM，避免上下文窗口被无关内容浪费
- **提前终止**：设置置信度阈值，Agent 确信已获得答案时提前结束循环，而非继续执行剩余计划步骤

**加分要点**：提到成本监控——为每个 Agent 任务记录 token 消耗和 API 调用次数，建立成本基线，当某个任务类型成本异常增长时自动告警。

## 六、Agent 应用开发

### 面试题 16：如何设计一个智能客服 Agent？

**考点**：垂直场景应用设计能力、端到端系统思维

**参考回答框架**：

智能客服 Agent 的设计可分为五个层次：

**意图理解层**：识别用户问题的真实意图，区分"咨询类"（产品信息、价格）和"操作类"（退款、修改订单）。对模糊输入做澄清追问，避免误解意图导致错误操作。

**知识检索层**：用 RAG 接入产品手册、FAQ、订单数据。关键优化点：对用户 query 做扩展改写（如"怎么退"→"退款政策/退货流程"）、用元数据过滤确保只检索用户有权查看的内容、对检索结果做相关性阈值过滤避免送噪声进 LLM。

**工具执行层**：封装订单查询、退款申请、物流跟踪等工具。每个工具要有严格的权限校验和操作审计，涉及资金/敏感信息的操作需要二次确认。

**对话管理层**：维护多轮对话状态，处理上下文切换（用户突然换话题）、意图澄清（"您是指 A 还是 B？"）、会话恢复（断线后重新连接）。

**兜底策略层**：当置信度低于阈值或遇到未训练场景时，优雅转人工，并携带完整的对话上下文和已尝试的解决方案。

**加分要点**：提到客服场景的评估特殊性——不仅要答对，还要考虑"是否让用户满意"（语气、响应速度、是否一次性解决问题），这些往往比准确率更重要。

### 面试题 17：Agent 在代码生成场景中的关键技术点是什么？

**考点**：代码类 Agent 的专项设计能力

**参考回答框架**：

代码生成 Agent 的关键技术点：

**上下文管理**：代码生成极度依赖上下文——文件结构、已有代码、依赖关系。需要将相关代码片段（而非整个仓库）精准送入 LLM，避免上下文窗口浪费。常用策略：先用代码结构摘要定位相关文件，再提取具体函数的上下文（定义+调用链）。

**执行验证**：生成的代码必须能运行且通过测试。建议为 Agent 配备沙箱执行环境（Docker 容器），让 Agent 能自行运行代码、查看报错、迭代修复。这是代码 Agent 区别于通用文本 Agent 的核心能力。

**工具链集成**：Agent 需要调用编译器、测试框架、代码分析工具（linter、类型检查）。这些工具的输出要结构化解析后反馈给 LLM，而非直接塞入原始终端输出。

**增量修改**：大多数场景不是从零写代码，而是修改现有代码。Agent 需要学会生成 diff 或 patch，而非完整文件重写，减少不必要的变更。

**加分要点**：提到 SWE-Agent 类系统的核心洞察——"让 Agent 像程序员一样工作"（读代码 → 理解 → 修改 → 测试 → 提交），而非"让 Agent 直接输出答案"。

### 面试题 18：Agent 的流式输出与用户体验如何设计？

**考点**：前端交互设计思维、用户体验意识

**参考回答框架**：

Agent 的 UX 设计核心是**让用户感知到进展，而非面对空白等待**。

**流式输出设计**：LLM 的生成结果应实时展示，用户看到文字逐字出现会比等待完整答案感觉更快。对于多步 Agent，应展示当前步骤的状态（"正在搜索相关知识..."、"正在调用计算工具..."），让用户理解系统在做什么。

**中间状态可视化**：Agent 的思考过程（Thought）、工具调用（Action）、观察结果（Observation）可以选择性展示。对于技术型用户，展示完整推理链增强可信度；对于普通用户，简化为进度条+关键节点提示。

**错误与重试的透明化**：当工具调用失败时，不要只显示"出错了"，而应说明"数据库查询超时，正在重试..."或"未找到相关信息，尝试用其他关键词搜索...」。这种透明化让用户感知到 Agent 在积极应对，而非僵死。

**人工介入点设计**：在关键决策节点（如执行资金操作、删除数据前）设置确认弹窗，让用户有掌控感。同时提供"停止"按钮，用户可随时中断 Agent 的自动执行。

**加分要点**：提到" perceived performance "（感知性能）比实际性能更重要——同样的延迟，有进度提示的体验远优于空白等待。
