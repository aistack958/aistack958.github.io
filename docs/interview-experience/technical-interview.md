# 技术面试实战技巧

技术面试不是考试，而是展示你思维方式和工程判断力的过程。本章提供具体的答题方法论，帮助你把已有的技术积累最大化地呈现给面试官。

## 如何讲清楚 Agent 系统设计

讲系统设计最大的误区是直接跳到架构图。面试官想听的不是"我用了 LangChain + ReAct"，而是你为什么选这条路径、经历了怎样的决策过程。

**推荐讲解框架——"四层递进法"**：

1. **需求层**：先花1-2分钟澄清需求边界。"这个Agent的目标用户是谁？核心场景是什么？对实时性、准确率、成本有什么约束？"主动提问比闷头设计更加分。
2. **架构层**：从整体到局部，先画出Agent的核心模块（感知→规划→执行→反馈），再说明每个模块的职责和边界。用白板或纸笔画出来，边画边讲，让面试官跟上你的思路。
3. **决策层**：说明关键设计决策及理由。"我选择 ReAct 而非纯规划式架构，因为这个场景需要Agent根据工具返回结果动态调整下一步行动。""我选择向量数据库而非关键词检索，因为用户查询通常是自然语言的模糊描述。"
4. **演进层**：讨论系统的可扩展性和可能遇到的问题。"随着工具数量增加，工具选择的准确率会下降，我计划引入工具路由的分层机制。""当前单Agent架构在复杂场景下可能力不从心，演进方向是引入多Agent分工协作。"

讲架构时务必控制节奏——每讲完一层停下来问面试官"这个方向您觉得合理吗？"，而不是一口气说完再等反馈。这种交互式讲解既展示了沟通能力，也避免你在一个错误方向上越走越远。

## 白板推导注意力机制

白板推导是算法岗面试的经典环节，AI Agent 岗位同样越来越多地考察这项能力。面试官想看的不只是你能写出公式，而是你对计算过程的完整理解。

**推导 Self-Attention 的推荐步骤**：

1. 先写出输入矩阵 X 的形状 (seq_len, d_model)，说明每个token有一个d_model维的向量表示
2. 写出三个投影矩阵 W_Q, W_K, W_V，推导 Q = XW_Q, K = XW_K, V = XW_V 的形状变化，每一步都标注维度
3. 计算注意力分数 Scores = QK^T / sqrt(d_k)，**重点解释为什么要除以 sqrt(d_k)**——这是防止点积过大导致Softmax梯度消失的缩放因子
4. 应用 Softmax 得到注意力权重，强调这是对每个查询token对所有键token的注意力分配
5. 计算输出 Attention(Q,K,V) = Softmax(Scores) * V，解释输出的含义——每个token的新表示是所有值token的加权混合

**关键加分点**：推导过程中主动解释每个步骤的直觉含义。比如在写 QK^T 时说"这一步的本质是计算每两个token之间的相关性，类似信息检索中查询和文档的匹配"。面试官会觉得你真正理解了而非只是记忆公式。

## 现场写 Agent 代码

现场编码题常见形式包括"写一个简单的ReAct Agent循环"、"实现一个工具调用的解析器"、"写一个多轮对话的状态管理器"等。

**编码策略**：

- **先写骨架再填细节**：先定义核心类和方法的签名，写好主循环结构，再逐步填充具体逻辑。这样面试官可以早期看到你的整体思路
- **边写边讲**：每写一个关键步骤就说一句"这里我处理的是工具返回结果的解析，如果解析失败我会让Agent重新选择工具"
- **处理边界情况**：主动处理异常情况——工具调用失败怎么办？LLM输出格式不对怎么办？循环超过最大步数怎么办？这些细节展示你的工程成熟度
- **保持代码可读性**：变量命名清晰、适当加注释、逻辑分块用函数封装，不要写面条代码

一个典型的 ReAct Agent 骨架只需约15行代码即可展示核心循环：接收用户输入→LLM生成思考+行动→解析行动调用工具→将工具结果加入上下文→循环直到生成最终答案。面试官看到这个骨架就知道你理解 Agent 的核心范式。

## 如何应对追问和压力面试

追问是技术面试的常态，面试官通过连续追问来测试你的知识边界和压力下的思维质量。常见追问模式：

- **深度追问**："你说的RAG召回率不高，具体是怎么量化的？你尝试了哪些优化手段？"——这是正面追问，展示你的实操深度
- **质疑追问**："你这个方案成本太高，在实际业务中根本不可行吧？"——这是压力测试，考验你面对质疑的反应
- **替代追问**："为什么不用微调而是RAG？微调在这个场景下不是更合适吗？"——考验你对多种方案的理解和判断力

**应对核心原则**：

1. **不慌不辩**：面对质疑不要急于反驳或认错。先确认面试官的假设："您说的成本高是指推理成本还是数据准备成本？"很多时候面试官的质疑基于特定假设，澄清假设后你可能发现你的方案其实没问题
2. **承认边界**：当追问到真正超出你知识范围的领域，诚实说"这部分我目前了解不够深入，我的理解是……但我需要进一步确认"，这比胡编乱造好得多
3. **给出替代方案**：当面试官指出你方案的缺陷时，不要只说"确实有问题"就停住，而是接着说"如果考虑这个约束，替代方案可以是……"，展示你的方案储备和灵活思维
4. **结构化回应**：追问时不要发散式回答，保持"问题定位→原因分析→解决方案→验证思路"的结构

压力面试的本质不是刁难，而是模拟真实工作中遇到质疑和约束的场景。展示你在压力下依然能结构化思考、灵活调整方案，比给出一个"完美答案"更有价值。

## 本章小结

技术面试四层应答法：
1. **需求澄清**：确认边界条件、明确约束
2. **架构设计**：分层阐述、权衡对比
3. **核心实现**：关键代码、复杂度分析
4. **扩展讨论**：容错、监控、演进

**现场编码**：先写骨架再填细节、边写边说、主动提边界情况。

**追问应对**：承认不知道 > 瞎猜；给出思考方向 > 直接放弃。
